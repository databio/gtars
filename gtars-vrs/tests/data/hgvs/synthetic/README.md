# Synthetic HGVS test corpus

A self-contained, hand-designed HGVS test set built on a synthetic genome
and a synthetic transcript table. Used by:

- `gtars-vrs/tests/test_hgvs_synthetic.rs` — layer 1 (VRS round-trip
  self-consistency) + layer 2 (coordinate transformation correctness)
- `gtars-vrs/tests/test_hgvs_synthetic_equiv.rs` — layer 3
  (N-way equivalence groups; see *Equivalence groups* below)
- `gtars-reftx/tests/test_synthetic_mapper.rs` — layer 2 only

## DO NOT hand-edit `cases.tsv`

`cases.tsv` is generated by `generate.py`. To change cases, edit the
generator and re-run:

```
PYTHONPATH=gtars-python/py_src python3 generate.py
```

`generate.py --check` regenerates to a tmpdir and diffs against the
checked-in files; CI uses this to catch fixture drift.

## Files

| File | What it is |
|---|---|
| `synthetic.fa` | One chromosome, `chr_synth`, length 30,000bp, seeded random ACGT |
| `synthetic.fa.fai` | samtools-style index for the FASTA (computed in Python) |
| `synthetic_transcripts.json` | cdot-shaped JSON for `TxStoreBuilder::ingest_cdot` |
| `refget_collection.json` | refget digest manifest (advisory; tests recompute) |
| `cases.tsv` | the test corpus (per-row cases for layers 1 + 2) |
| `equivalence_groups.tsv` | N-way equivalence groups for layer 3 |

The genome seed is hard-coded in `generate.py` as `GENOME_SEED = 0xDA51`.
Bumping `FIXTURE_VERSION` (also in `generate.py`) signals a deliberate
corpus change to anyone reviewing diffs.

## Synthetic transcripts

All on `chr_synth`. 1-based inclusive coordinates (cdot convention).

| Accession | Strand | Type | Exons (genomic) | CDS | Notes |
|---|---|---|---|---|---|
| `NM_synth_simple` | + | coding | `[(1001,1300)]` | `(1051,1249)` | single exon, 50bp 5'UTR, 199bp CDS, 51bp 3'UTR |
| `NM_synth_fwd` | + | coding | `[(2001,2100),(2301,2450),(2701,2850),(3101,3200)]` | `(2051,3150)` | 4 exons, 5'UTR in exon 1, CDS spans all 4 |
| `NM_synth_rev` | - | coding | `[(5001,5100),(5301,5450),(5701,5850),(6101,6200)]` | `(5051,6150)` | mirror of `NM_synth_fwd` on minus strand |
| `NR_synth_nc` | + | noncoding | `[(8001,8050),(8201,8300),(8501,8550)]` | none | for `n.` variants |
| `NM_synth_tiny` | + | coding | `[(9001,9010),(9015,9030),(9035,9044)]` | `(9003,9042)` | tiny exons + 4bp introns to stress edge cases |

## Reverse-strand transcript orientation

For minus-strand `NM_synth_rev` with exons `[(5001,5100),(5301,5450),(5701,5850),(6101,6200)]`:

- Transcript orientation runs 3' (genomic right) → 5' (genomic left) in
  genomic coordinates, but 5'→3' in transcript coordinates.
- Transcript position 1 = genomic 6200; transcript position 100 = genomic
  6101; transcript position 101 = genomic 5850; etc.
- Walk exons in *reverse genomic order* (rightmost first), accumulating
  transcript length. For each exon `(g_start, g_end)` of length `L`,
  transcript positions `[acc+1 .. acc+L]` correspond to genomic positions
  `[g_end .. g_start]` (decreasing).
- The displayed HGVS allele on a minus-strand transcript is the
  reverse-complement of the genomic allele. So `NM_synth_rev:c.10A>T`
  corresponds to genomic `chr_synth:g.<pos>T>A`.
- `expected_ref` and `expected_alt` in `cases.tsv` are in **forward-strand
  genomic** orientation, matching what `gtars_vrs::vcf` and the bridge
  produce.

## Two-layer test design

**Layer 1 — VRS round-trip self-consistency.** For each non-error case,
the test computes a VRS ID two ways:

1. HGVS path: `gtars_vrs::hgvs::hgvs_str_to_vrs_id`
2. VCF path: `(chrom, pos, ref, alt)` → normalize → digest

Equality demonstrates the HGVS bridge agrees with the canonical VCF→VRS
pipeline. `expected_vrs_id` in `cases.tsv` is the precomputed VCF-path
result. **Layer 1 is a self-consistency check, not an independent
oracle** — both paths share the same Rust normalize+digest code.

**Layer 2 — coordinate transformation correctness.** For each non-genomic
case (`c.` / `n.`), the *generator* records `expected_pos` by walking the
synthetic transcript exon table directly with its own brute-force
algorithm. The mapper under test (`gtars_reftx::CoordinateMapper`)
computes the same coordinate via its production algorithm. The two paths
share only the fixture data; the algorithms are independent. A strand
bug, off-by-one in CDS offset, mishandled negative intronic offset, or
5'UTR edge bug in the production mapper causes the layer-2 assertion to
fail.

This is the only place in the gtars test tree where we have a genuinely
independent oracle for `c.→g.` mapping.

## Failure attribution

When both layers fail on the same row, the test reports it as a
**layer-2** failure only. Reasoning: if the bridge cannot map `c.` to
`g.` correctly, its VRS ID is meaningless and a layer-1 mismatch is a
derived symptom, not a separate bug. The harness groups failures by
layer so the developer immediately sees "5 mapper bugs and 0 bridge
bugs" rather than 10 confused failures.

## Equivalence groups

`equivalence_groups.tsv` (plan 12) extends the per-row corpus with **N-way
equivalence groups**: sets of expressions across forms (g., c., n., VCF
tuple) that all describe the same biological variant and MUST collapse to
the same VRS ID. This catches *representation drift* — bugs that the
per-row layers cannot see, such as the bridge producing one VRS ID for
`g.X` and a different one for `c.Y` when both describe the same allele.

Schema (six columns, tab-separated):

| column | meaning |
|---|---|
| `group_id` | stable label, e.g. `eq_fwd_sub_at_cds_start_001` |
| `member_kind` | one of `hgvs_g`, `hgvs_c`, `hgvs_n`, `vcf` |
| `expression` | literal HGVS string, OR for `vcf` a colon-tuple `chrom:pos:ref:alt` |
| `transcript` | tx accession (only meaningful for `hgvs_c` / `hgvs_n`; empty otherwise) |
| `expected_vrs_id` | the canonical VRS ID for the whole group (repeated identically on every row of the group) |
| `notes` | free text |

Each group asserts: every member's HGVS→VRS (or VCF→VRS) computation
yields the same VRS ID, and that ID equals `expected_vrs_id`.

`expected_vrs_id` is computed once per group **at fixture-generation time**
via the canonical VCF→VRS path (`gtars.vrs.normalize_allele` +
`gtars.vrs.vrs_id`). The harness does NOT recompute the canonical VRS ID —
that would re-use the very logic under test. The harness merely runs each
member through its respective bridge entry point and compares.

### Failure attribution

When a group fails, the harness does not panic on the first diverging
member. Instead, it buckets all members by computed VRS ID (or by error
message) and prints a structured report:

```
EQUIVALENCE FAILURE: group_id = eq_fwd_del_len1_001
  expected_vrs_id (from TSV): ga4gh:VA.sMhqHWChcUpaiCuFyL3EAsfLhU0E-chn
  Bucket 1 (id=ga4gh:VA.2oxGkSzdfYGOrpTLazF628ixkVVNdEM8):
    - hgvs_c[NM_synth_fwd:c.11del]
  Bucket 2 (id=ga4gh:VA.sMhqHWChcUpaiCuFyL3EAsfLhU0E-chn):
    - hgvs_g[chr_synth:g.2060del]
    - vcf[chr_synth:2060:A:]
```

All groups are run to completion; one panic at the end summarizes every
failing group. This guarantees a single test run gives the developer full
attribution: which forms agree, which diverge, and the exact bucketing.

### Cross-link with `cases.tsv`

Each `expected_vrs_id` in `equivalence_groups.tsv` is also present in
`cases.tsv` — the equivalence picks are deliberately a curated subset of
the same biological tuples used as per-row cases. The harness asserts
this invariant; drift between the two files trips a sanity check.

### Regeneration

```
PYTHONPATH=gtars-python/py_src python3 generate.py            # rewrites both TSVs
PYTHONPATH=gtars-python/py_src python3 generate.py --check    # diff against committed
```

`--check` validates both `cases.tsv` and `equivalence_groups.tsv`.

### Scope limitations

Documented explicitly so future readers don't assume gaps are bugs:

- **No MANE Select equivalence.** The synthetic transcripts have no MANE
  Select assignment, so `gene-symbol → MANE` equivalence cannot be tested
  here. That path is exercised by the bio-daas-variant integration tests
  against real data.
- **No protein expressions.** `p.` form is out of scope.
- **No alternate normalization forms.** Left- vs right-aligned `dup`
  variants are not separately exercised.
- **No 4-way (g + c + n + VCF) or 5-way (g + c.fwd + c.rev + n + VCF)
  groups.** The current synthetic transcript layout has no genomic region
  that is covered by both a coding transcript and `NR_synth_nc`, and no
  region overlapped by both `NM_synth_fwd` and `NM_synth_rev`. Maximum
  multiplicity is therefore 3-way (g + one transcript form + VCF). If the
  layout grows an overlapping pair, extend `_EQ_PICKS_RAW` and
  `generate_equivalence_groups()` to emit higher-arity groups.

## Non-circularity rule (CRITICAL)

```
THE GENERATOR MUST NOT IMPORT, CALL, OR RE-IMPLEMENT BY COPYING ANY CODE
FROM gtars_reftx::CoordinateMapper OR gtars_vrs::hgvs::bridge.

It walks exon tables directly to compute expected genomic coordinates.
Sharing coordinate-transformation code with the Rust crates under test
turns layer-2 of the synthetic test set into a tautology and silently
masks real bugs. If you find yourself wanting to import a Rust helper
to "avoid duplication" -- STOP. Duplication is the entire point.

Layer-1 VRS-ID generation MAY use the gtars PyO3 module because layer 1
is documented as a self-consistency check, not an independent oracle.
The same exemption applies to layer-3 `expected_vrs_id` in
`equivalence_groups.tsv`: it is computed via the canonical VCF→VRS path
in the generator, never recomputed in the harness.
```

This docstring header is duplicated verbatim in `generate.py` and a CI
guard greps for it.
